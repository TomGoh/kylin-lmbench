# pKVM overhead on Phytium FTC862 ‚Äî day 1 findings

**Date**: 2026-06-01
**Target machine**: N90 (Phytium FTC862, 8 cores, 2 cluster, KylinOS V11)
**Kernel**: 6.6.0-52-generic (Kylin pKVM build, `kvm-arm.mode=protected`)
**Repo state**: `kylin-lmbench` @ commit `79d5922`, plus local fixes from this session

## TL;DR

After four rounds of methodology alignment, we now have **MAD% < 2% paper-grade
N=5 medians on both host and guest** (CONFIG.test, cpu0 pinned, 2.1 GHz locked).
Two distinct signals emerged:

1. **Guest is ~20% faster than host on the syscall fastpath** ‚Äî not a
   real pKVM effect; root cause is that KYSEC userspace (`ksaf-policy-init`,
   `auditd`, `/sys/fs/box`, BPF programs) **does not activate in the guest**,
   so guest's LSM hooks take a short path while host's run the full policy.
2. **Guest is consistently ~+3.5% slower on stage-2-touching operations**
   (fork, mprotect, IPC, minor pagefault) ‚Äî this is the **real pKVM tax**
   when neutralizing the KYSEC asymmetry.

`tlb effective = 48 pages` identical on both sides ‚Äî at single-process,
small-working-set granularity pKVM does not contend for hardware TLB.

## Root-cause analysis

### Why guest is faster on the syscall fastpath: KYSEC asymmetry

Inventory dumped from inside the guest at benchmark time (full output in
`results/n90-day1/guest-N5-bootlog.txt`, line 60+):

| Mechanism | Host | Guest | Effect |
|-----------|------|-------|--------|
| `/sys/kernel/security/lsm` | `capability,kycp,ksaf` | `capability,kycp,ksaf` | ‚úì identical |
| `ksaf-devctl-sync-daemon` | active | active | ‚úì |
| `ksaf-label-manager` | active | active | ‚úì |
| `ksaf-policy-init.service` | active | **inactive** | ‚öÝÔ∏è policy not loaded |
| `kysec-scene-init.service` | active | active | ‚úì |
| `kysec2-kbox-load.service` | active | **inactive** | ‚öÝÔ∏è kbox module absent |
| `/sys/fs/box` | exists | **does not exist** | ‚öÝÔ∏è sandbox infra absent |
| `auditd.service` | active (pid 882) | **failed** | ‚öÝÔ∏è no audit consumer |
| audit `backlog` / `lost` | 0 / 42 | 0 / 211 | guest losing records |
| audit rules | 1 (Desktop file ops) | 0 | guest has no rules |
| BPF programs loaded | many (cgroup_skb / cgroup_device) | **0** | guest has no systemd BPF |

The LSM module is loaded on both sides, but the userspace policy daemons
that populate the in-kernel policy tables do not initialize in the guest
(`ksaf-policy-init` is `inactive`). Result: `ksaf` LSM hooks fire on both
sides but take a short path in the guest (no rules to evaluate). The 60+
cycle delta per syscall is consistent across `lat_syscall null/read/write/
stat/open/fstat/lat_sig install` and matches the magnitude of "KYSEC active
vs not."

`lat_syscall null` cycle accounting:
- host: 0.1524 ¬µs √ó 2.1 GHz ‚âà **320 cycles**
- guest: 0.1125 ¬µs √ó 2.1 GHz ‚âà **236 cycles**
- delta: **~84 cycles per syscall** charged to active KYSEC userspace

### Why guest is slower on the +3.5% group: real pKVM trap cost

For benchmarks that touch stage-2 page tables or trap to EL2 hypervisor:

- `lat_sig prot` (mprotect) ‚Äî modifying stage-1 PTE triggers stage-2 sync
- `lat_pagefault minor` ‚Äî first-touch faults need stage-2 PTE allocation in
  protected mode (though our guest is `pkvm-guest-np`, non-protected, so
  the cost is purely hypervisor sched + IRQ overhead)
- `lat_pipe rt`, `lat_unix rt` ‚Äî IPC requires multiple syscalls + signal
  deliveries which traverse the hypervisor on every guest entry/exit
- `lat_proc fork` ‚Äî fork copies the mm_struct; in pKVM mode each new mapping
  in the new mm requires stage-2 PTE setup

All five land in the **+2.8% to +3.5% band** with MAD% under 2%. The signal
is sharp and reproducible.

Note: `lat_proc fork +3.5%` (4.3 ¬µs absolute) is **much lower** than
reported in early pKVM literature (10-30%). This suggests the Kylin
6.6.0-52-pkvm10 build has efficient stage-2 fault handling, or our test
setup (single-vCPU, no real I/O, single iteration of fork+exit) does not
expose the worst-case path. Worth retesting under multi-vCPU scenarios.

## Caveat: KYSEC asymmetry is a confound on group B too

Because the host has KYSEC actively running while the guest does not, the
"+3.5% pKVM tax" measured here is computed against an already-30%-faster
guest baseline. The TRUE pKVM tax ‚Äî measured with KYSEC equally active on
both sides ‚Äî could be slightly higher. Two options to clean this up:

- **A. Get `ksaf-policy-init` to succeed in the guest.** Requires debugging
  why the service fails (probably missing `/dev/box` device node and
  `kysec2-kbox-load` failing). High effort; gives a fully aligned
  comparison.
- **B. Re-run with KYSEC disabled on both sides.** Boot host with
  `lsm= security=` removed (requires reboot); guest cmdline omit the LSM
  flags. Lower effort; gives a "pure hypervisor" measurement.

Option B is faster and the resulting data has a clearer narrative ("if
KYSEC is disabled, here's what pKVM costs"); option A is required for
production-realistic numbers.

## Open questions

- **Does the +3.5% scale with the workload?** Repeat with CONFIG.host (full
  lmbench, larger memory sweep, multi-stride lat_mem_rd, BENCHMARK_FILE on
  virtio-blk).
- **Does TLB pressure rise under larger working sets?** lat_mem_rd at >> L2
  size should reveal stage-2 walk contention if it exists.
- **What does pKVM protected guest (`--protected-vm-without-firmware`) cost
  beyond `pkvm-guest-np`?** That tests the bounce-buffer / share-unshare
  path which we haven't measured yet.
- **Is `lat_proc fork +3.5%` representative of fork-heavy workloads?**
  Single fork+exit per iter; high-rate fork might amortize the trap cost
  differently.

## What goes into the paper

A clean Methods + Results narrative for this dataset:

1. **Methods**: list the variables we held constant (kernel binary, cmdline
   modulo virt-mode line, userspace via ostree clone, locked CPU freq,
   THP=never, ASLR=0, LSM stack identical at module level). Acknowledge
   the KYSEC-policy state asymmetry explicitly.
2. **Results**: present Group A (negative deltas) as evidence of the KYSEC
   userspace overhead on bare host, and Group B (positive deltas) as the
   pKVM tax for stage-2 / hyp-trap operations.
3. **Discussion**: 3.5% is small enough to suggest pKVM has matured for
   production use on this hardware; the larger 20% KYSEC delta is a
   separate, orthogonal cost that future work could address by aligning
   policy state across host and guest.
