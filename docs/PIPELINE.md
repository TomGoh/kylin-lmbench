# Pipeline internals

How the scripts fit together and what they each do.

## End-to-end flow

```
              ┌─────────────────────────────────────────────────────┐
              │            bench.sh  (entry point)                  │
              └──────────┬──────────────────────────────────────────┘
                         │
              ┌──────────▼───────────┐
              │ prepare-host.sh      │   sudo: freq lock, THP off, ASLR off,
              │  (skippable, sudo)   │   deep-idle off, rpcbind start
              └──────────┬───────────┘
                         │
              ┌──────────▼───────────────────────────────────────┐
              │ for core in $CORES:                              │
              │   for i in 1..$ITERS:                            │
              │     ( cd bin/aarch64-Linux                       │
              │       taskset -c $core                           │
              │       bash scripts/lmbench $CONFIG               │
              │     ) 2> results/$env-cpu$core-iter$i.txt        │
              └──────────┬───────────────────────────────────────┘
                         │
              ┌──────────▼───────────────────────────────────────┐
              │ parse-lmbench.py iter*.txt > $env-cpu$core.csv   │
              └──────────────────────────────────────────────────┘
```

## Files and what they own

### `bench.sh` — primary driver
Sole entry point for a real run. Reads env vars `ENV_TAG`, `CORES`, `ITERS`,
`CONFIG`. Owns:
- calling `prepare-host.sh` (unless `--no-prep`)
- looping over cores and iterations
- correctly cd'ing into `bin/aarch64-Linux/` before invoking `scripts/lmbench`
  (PATH=. in the lmbench script requires this)
- wrapping every invocation with `taskset -c $core`
- driving the parser at the end
- writing a per-run `<env>-<timestamp>-summary.txt` with kernel/cmdline/freq info

### `prepare-host.sh` — environment setup
Idempotent. Self-elevates via `exec sudo -E` if not run as root. Knobs (env
vars): `TARGET_KHZ`, `DISABLE_THP`, `DISABLE_ASLR`, `DISABLE_DEEP_IDLE`. The
`exec sudo -E` pattern means the password is asked once at the top of `bench.sh`
and the elevated process inherits everything via `-E`.

The freq-locking code is careful about ordering: if the target frequency is
**lower** than the current `scaling_min_freq`, it lowers `min` first (otherwise
the kernel rejects `max < min` as a transient state). If **higher**, it raises
`max` first.

`rpcbind` is started best-effort. Without it, `lat_rpc` emits a few error lines
to stderr instead of valid timings; the parser skips those silently.

### `parse-lmbench.py` — text → CSV
Reads lmbench's stderr report (which contains all benchmark data, despite the
name). It's a small state machine:

- **Single-line patterns** match things like `Simple syscall: 0.1697 microseconds`
  and emit immediately. Each pattern is in `SINGLE_LINE` with explicit (bench,
  variant, unit) — easy to add new ones.
- **Mode banners** (literal text like `Memory load latency`, `"mappings`,
  `Socket bandwidth using localhost`) flip parser state. Subsequent table-row
  lines are interpreted under that mode.
- **Sub-headers** within a mode (`"size=0k ovr=0.71`, `"stride=128`) further
  qualify the rows (carry context like ctx working-set size or stride).
- **Blank lines** reset all mode state — lmbench uses blank lines to delimit
  tables.
- **Skip patterns** (RPC errors, metadata blocks) are dropped silently.
- **Anything else** is logged to stderr as `UNPARSED [file]: <line>`, so the
  parser can be extended iteratively without losing data.

Filename convention: `<env>-cpu<N>-iter<I>.txt`. The script derives `env`,
`core`, `iter` from the filename unless `--env/--core/--iter` are passed.

Output CSV columns: `env, core, iter, bench, variant, value, unit`. Long
format, one row per measurement. Aggregation (median/MAD across iters) is
intentionally not done here — keep raw and aggregate downstream.

### `configs/CONFIG.*`
Hand-written CONFIG files, **not** generated by upstream's interactive
`scripts/config-run`. The interactive flow asks ~30 questions and tends to
auto-detect things wrong (e.g. `MHZ=4048` on an ARM machine because it reads
the generic timer). Hand-writing avoids this entire failure mode.

`CONFIG.template` is the annotated reference — every field has a comment.
Per-environment configs are spartan (just values) and link back to the template
in the first comment line.

The two `BENCHMARK_*` patterns to understand:
- `BENCHMARK_OS=YES` is a master switch that enables many subsystems including
  network and FS. **We never set this** — we enable per-category flags instead,
  so the set of benchmarks actually run is explicit.
- `BENCHMARK_HARDWARE=YES` is similar for memory/hardware. We use the more
  targeted `BENCHMARK_MEM=YES` which gives lat_mem_rd + tlb + par_mem + stream
  without the full bw_mem flavor explosion.

### Smoke test
For a 90-second pipeline + parser validation (no full bench), use the
minimal `CONFIG.test` subset with `bench.sh`:

```bash
CORES=0 ITERS=1 CONFIG=configs/CONFIG.test ENV_TAG=smoke ./bench.sh
```

Output ends up in `results/smoke-cpu0.csv` like any other run, so the
parser path gets exercised end-to-end.

> An older pilot-era trio (`run-pilot.sh` / `run-mem.sh` / `bench-host.sh`)
> bypassed `scripts/lmbench` and ran a hand-picked subset directly. Those
> scripts were removed during the post-paper cleanup — `bench.sh` is the
> only driver now.

## Adding a new environment

```bash
# 1. write a CONFIG (start from CONFIG.template)
cp configs/CONFIG.template configs/CONFIG.my-new-env
$EDITOR configs/CONFIG.my-new-env       # fill in MHZ, PROCESSORS, etc.

# 2. invoke bench.sh
CONFIG=configs/CONFIG.my-new-env \
  ENV_TAG=my-new-env \
  CORES=0,1 \
  ITERS=10 \
  ./bench.sh
```

In a restricted environment where you can't (or don't want to) write to
`cpufreq` — for example a remote box you don't own, or a container — pass
`--no-prep`:

```bash
ENV_TAG=my-restricted-env CONFIG=configs/CONFIG.my-restricted-env \
  ./bench.sh --no-prep
```

## Adding a new benchmark to the parser

1. Run lmbench with your benchmark enabled, copy the output line(s) verbatim.
2. If single-line: add a regex entry to `SINGLE_LINE` in `parse-lmbench.py`.
3. If table: add a banner string to `MODE_BANNERS` and (if the row format isn't
   `<x> <y>`) add a handler in the main loop.
4. Re-parse a sample report, watch for `UNPARSED [...]` entries on stderr.

## Why we don't use `make rerun`

`make rerun` requires the CONFIG file to live at `bin/aarch64-Linux/CONFIG.$USER`
and runs from that directory. We invoke `scripts/lmbench <our-config-path>`
directly instead, which:
- lets us keep CONFIGs in `configs/` (organized, version-controlled)
- skips the `make config` interactive step
- gives us explicit control over CWD (must be `bin/aarch64-Linux/`) and PATH

The constraint to know about: `scripts/lmbench` puts `.` first in `PATH` to
find the lmbench tools, so the invoker **must `cd` into `bin/aarch64-Linux/`
before exec**. `bench.sh` does this with `( cd $BIN_DIR && ... )` subshells so
that the parent shell's CWD is not perturbed.
